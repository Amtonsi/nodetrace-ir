# Участие в разработке NodeTrace IR

Спасибо за помощь. Проект принимает исправления, новые defensive-коллекторы, улучшения модели доказательств, тесты и документацию.

## Основные принципы

Любое изменение должно сохранять четыре свойства:

1. **Безопасность:** подозрительный объект не исполняется, не импортируется и не передаётся внешнему сервису.
2. **Проверяемость:** каждый факт содержит источник и ссылку на него, а каждая корреляция — уверенность и текстовое обоснование.
3. **Честное покрытие:** недоступность, усечение и неоднозначность оформляются как `GapDraft`, а не скрываются.
4. **Локальность:** по умолчанию данные остаются на узле; новая сетевая интеграция требует отдельного явного дизайна, согласия пользователя и threat review.

Не добавляйте функции скрытого доступа, закрепления, сбора секретов, обхода средств защиты, исполнения образцов или автоматического воздействия на исследуемый узел.

## Настройка среды

```powershell
git clone <URL-репозитория>
cd nodetrace-ir
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -e . pytest pyinstaller
.\scripts\run_tests.ps1
```

Поддерживаемая базовая версия — Python 3.11+. Runtime-код должен по возможности оставаться совместимым со стандартной библиотекой Windows Python.

## Рабочий процесс

1. Создайте issue с наблюдаемым поведением, ожидаемым результатом и границами задачи.
2. Создайте отдельную ветку от актуальной `main`.
3. Сделайте небольшой логически цельный change set.
4. Добавьте или обновите тесты.
5. Запустите `scripts/run_tests.ps1`.
6. Обновите README или документы в `docs/`, если меняются интерфейс, источники, формат данных или ограничения.
7. Откройте pull request с описанием риска, проверки и влияния на доказательность.

Не включайте в commit:

- `.sqlite3`, реальные отчёты и каталоги кейсов;
- журналы, дампы, персональные данные и сетевые идентификаторы реальных узлов;
- подозрительные/вредоносные бинарники;
- архивы, базы и бинарники AVZ, а также WIM/CAB/MSI/ISO и другие payload WinPE/ADK;
- `build/`, `dist/`, `*.spec`, `__pycache__` и локальное окружение;
- ключи подписи, токены и иные секреты.

## Требования к коллектору

Коллектор реализует контракт из `nodetrace_ir/contracts.py`:

```python
class Collector(Protocol):
    name: str
    display_name: str

    def collect(self, context: CollectionContext) -> CollectorResult:
        ...
```

Новый коллектор должен:

- иметь стабильное уникальное `name`;
- выполнять только ограниченные и документированные запросы без ремедиации или изменения конфигурации;
- использовать `context.cancel_event` для кооперативной отмены;
- задавать конечные лимиты и таймауты;
- возвращать `EvidenceDraft`, `RelationDraft` и `GapDraft`, а не писать в БД напрямую;
- передавать подозрительные пути через аргументы/переменные окружения, не интерполируя их в команды;
- сохранять исходную запись в `raw`, нормализованные поля — в `properties`;
- присваивать устойчивый `stable_key`, чтобы повторный сбор не плодил дубликаты;
- не повышать уверенность только из-за временной близости или совпадения имени;
- добавлять gap при отказе источника, нехватке прав, таймауте, лимите или отключённой телеметрии;
- иметь unit-тесты для нормального результата, отказа, отмены и неоднозначной корреляции;
- быть описан в `docs/COLLECTORS.md`.

PowerShell-запросы должны быть фиксированными, выполняться без профиля и интерактива, возвращать ограниченный JSON и не использовать `Invoke-Expression`. Непроверенные значения не должны становиться частью исходного текста PowerShell.

## Модель доказательств

При выборе `confidence` ориентируйтесь на [docs/EVIDENCE_MODEL.md](docs/EVIDENCE_MODEL.md):

- `high` — прямое значение из конкретного источника;
- `medium` — обоснованная корреляция с известной неоднозначностью;
- `low` — контекстная гипотеза, требующая независимого подтверждения.

`severity` — приоритет проверки, а не вердикт о вредоносности. Нельзя утверждать, что объект заражён, только потому что он без подписи, встречается рядом по времени или присутствует в Prefetch.

## Стиль и тестирование

- Используйте type hints и небольшие чистые функции там, где это упрощает проверку.
- Поддерживайте UTF-8 для русского интерфейса и экспортов.
- Не добавляйте обязательную runtime-зависимость без обоснования.
- Изолируйте ошибки одного источника: сбой коллектора не должен уничтожать результаты остальных.
- Проверяйте как минимум `python -m pytest -q` и `python run_nodetrace_ir.py --smoke-test`.
- Для изменений экспортера проверьте содержимое ZIP, `manifest.json` и `SHA256SUMS.txt`.
- Для GUI-изменений приложите скриншоты, но не используйте данные реального расследования.

## Pull request checklist

- [ ] Изменение относится к defensive incident response.
- [ ] Подозрительный файл нигде не исполняется.
- [ ] Нет неявной отправки данных или новых сетевых обращений.
- [ ] Источники, уверенность, ограничения и gaps задокументированы.
- [ ] Тесты добавлены и проходят.
- [ ] В diff нет реальных кейсов, бинарников и секретов.
- [ ] Пользовательская документация обновлена.

Уязвимости не следует раскрывать через обычный PR или issue — используйте [SECURITY.md](SECURITY.md).
